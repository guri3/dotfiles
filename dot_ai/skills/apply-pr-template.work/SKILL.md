# apply-pr-template.work

## 目的

既存 PR の body を、リポジトリで定義された PR テンプレートのセクション構造に沿うよう整形・更新するためのスキルである。
テンプレートに沿っていない PR を一貫した形式に揃えることで、レビュアーの認知負荷を下げ、必須項目の記入漏れを防ぐ。

## 使うタイミング

- 「PR テンプレート適用して」「テンプレートに合わせて」「PR 本文を直して」「テンプレートが反映されてない」といった明示的な依頼があったとき
- PR 作成後に body がテンプレートのセクション構造に沿っていないと気付いたとき
- create-pr スキルで PR を作成した直後にテンプレートと突き合わせて整形したいとき
- PR レビュー時にテンプレート不備を発見した場面

## 手順

### 1. 対象 PR の特定

引数や直近の文脈から PR 番号または URL を取得する。明示的な指定がない場合は現在のブランチに紐付く PR を探す。

```bash
# 現在ブランチの PR を取得
gh pr view --json number,body,url
```

PR 番号が判明したら、以降の `gh` コマンドで使い回す。

### 2. リポジトリのテンプレート取得

以下の探索順でテンプレートファイルを特定する。

1. `.github/pull_request_template.md`
2. `.github/PULL_REQUEST_TEMPLATE.md`
3. `.github/PULL_REQUEST_TEMPLATE/*.md`（複数ある場合はファイル名から推測するか、ユーザーに選択を仰ぐ）
4. リポジトリ root の `pull_request_template.md`

いずれも見つからない場合はユーザーに確認し、整形作業を中断する。

### 3. 既存 body のパース・マッピング

- 現在の PR body から見出し（`#` / `##` / `###`）を抽出する
- 各セクションの本文をテンプレート側の見出しと突き合わせる
- 見出し名が完全一致しない場合は意味的な近さ（例: 「概要」と「Summary」）でマッピングし、判断に迷うときはユーザーに確認する

### 4. テンプレート構造に整形

- テンプレートのセクション順に並び替える
- テンプレートにあるが body に存在しない `[必須]` セクションは、空でも見出しを残す
- body にあるがテンプレートに対応しないセクションは末尾の「その他」セクションにまとめる、もしくはユーザーに残すか削除するかを確認する
- テンプレート内のコメント行（`<!-- -->`）や説明文は、必要に応じて保持する
- テンプレートが「不要な項目は削除して利用してください」と明記している場合、空セクションは削除してよい

### 5. diff 確認

整形前後の body の diff をチャット上に表示し、ユーザーの承認を得てから次のステップへ進む。

承認なしに `gh pr edit` を実行してはならない。

### 6. PR body 更新

承認後、整形済み body を一時ファイルに書き出し `--body-file` で更新する。バックスラッシュやバッククオートのエスケープ事故を避けるため `--body` ではなく `--body-file` を使う。

```bash
# 整形済み body を一時ファイルに書き出す
TMP_BODY=$(mktemp)
cat <<'EOF' > "$TMP_BODY"
<整形後の PR body>
EOF

# PR body を更新
gh pr edit <PR番号> --body-file "$TMP_BODY"

# 後始末
rm "$TMP_BODY"
```

更新後は `gh pr view <PR番号> --json body` で反映を確認する。

## 注意

- テンプレートに「不要な項目は削除して利用してください」と注記がある場合、空セクションは削除してよい
- `[必須]` 印のあるセクションは内容が空でも見出しを残す
- バックスラッシュやインラインコードのエスケープ事故を避けるため、heredoc + `gh pr edit --body-file` の組み合わせを必ず使う（`--body` への直接渡しは避ける）
- diff 表示と承認のステップは省略しない。意図せず本文が消える事故を防ぐためにユーザー承認を必須とする
- 既存 body に PR レビューでのやり取りや手書きメモが含まれている場合、機械的に削除せず「その他」セクション等に退避する判断をユーザーに仰ぐ
