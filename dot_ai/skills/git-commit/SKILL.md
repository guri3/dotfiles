# git commit and push

これまでの作業内容に適切なコミットメッセージをつけてcommit、pushする。

## 手順

1. **変更状況の確認**
   - `git status` で未コミットの変更とuntrackedファイルを確認する
   - `git diff` でtracked fileの変更内容を確認する
   - `git diff` では新規(untracked)ファイルの中身は見えない。untrackedがある場合は Read ツールでそれぞれの中身を確認する
   - 変更がない場合はその旨を伝えて終了する

2. **ステージング**
   - コミットすべき変更を選択的に `git add <ファイル名>` でステージする
   - `git add -A` や `git add .` は意図しないファイルを含む恐れがあるため使わない
   - **対象の限定指定**: ユーザーが対象を指定した場合（例: 「◯◯の変更だけ」「この2ファイルだけ」「△△以外を」）、コミット対象をそのスコープに厳密に限定する
     - 指定に含まれない変更はステージせず、コミット後も未ステージのまま残す。残りを勝手に別コミットにしない
     - 指定が曖昧でどのファイルが該当するか確信が持てない場合は、`git status` から該当しうる候補を示して確認する
     - 限定指定はコミット範囲のみに適用し、pushは通常どおり行う
   - `git status` の出力を目視で確認し、以下に該当するファイルが含まれていれば **機密の疑いがあるファイルがある旨をユーザーに一言報告し、既定ではそのファイルをステージせずに残りの変更だけでコミットを続行する**
     - 機密情報・認証情報: 名前に `.env`、`secret`、`credentials`、`.pem`、`.key`、`token` を含むファイル
     - エディタ/IDEのローカル設定: `.idea/`、`.vscode/`、`.DS_Store` 等
     - 想定外の生成物（ビルド成果物・ログ・一時ファイル）と思しきファイル
   - `.gitignore` によって既に無視されているファイルは報告不要
   - ユーザー応答を同期的に待つ必要はない。後からユーザーが明示的に「含めてよい」と応答した場合にのみ、該当ファイルを追加でステージして次のコミットに含める

3. **コミット**
   - 変更内容に基づいて適切なコミットメッセージを作成する
   - Conventional Commits に従い、以下のプレフィックスを付与する
     - `feat:` 新機能の追加
     - `fix:` バグ修正
     - `docs:` ドキュメントの更新
     - `style:` 空白、フォーマット、セミコロン追加など
     - `refactor:` 仕様に影響がないコード改善（リファクタ）
     - `perf:` パフォーマンス向上関連
     - `test:` テストの修正
     - `chore:` ビルド、補助ツール、ライブラリ関連の修正
   - メッセージは日本語で記述する
   - 例: `feat: ユーザー登録APIの実装`
   - **コミット分割の判断**: 以下に1つでも該当すれば原則として別コミットに分ける
     - 該当する Conventional Commits プレフィックスが異なる（例: `feat` と `fix` が混在している）
     - 変更対象のファイル群が独立しており、片方だけを revert / cherry-pick したいケースが想定できる
     - ただし、同一目的に属する関連変更（実装＋そのテスト、機能追加＋その機能に必要な型定義の追加 等）は1コミットにまとめて良い
   - **コミット順序**: 依存関係がある場合は依存される側を先にコミットする。依存関係がない場合は `fix:` → `feat:` の順（既存バグの修正を先に置くことで履歴が素直になる）
   - **コミットメッセージの渡し方**:
     - 1行メッセージは `-m` オプションに直接文字列を渡す。例: `git commit -m "feat: ユーザー登録APIの実装"`
     - 本文を付ける場合は `-m` を複数回使う。例: `git commit -m "feat: ユーザー登録APIの実装" -m "バリデーション追加"`
     - `-m "$(cat <<EOF ... EOF)"` のようなコマンド置換は使わない（CLAUDE.mdルール準拠）

4. **プッシュ**
   - upstreamの有無を `git rev-parse --abbrev-ref --symbolic-full-name @{u} 2>/dev/null` で確認する
     - 出力があれば既にupstreamがあるので `git push`
     - 出力が空 / 失敗すればupstream未設定なので `git push -u origin $(git branch --show-current)`
   - 複数コミットが積まれている場合でも push は最後にまとめて1回で良い

## ルール

- 勝手に修正が必要だと判断して（lintエラー修正やリファクタ等を）変更内容に加えないこと。既にある変更のみをcommitする。
