# 呟き駆動の日報生成（nippo）

セッション中に記録した呟き（`#t` で残したメモ）と、その日の全セッションの作業履歴を統合し、JST の時系列タイムラインを備えた日報を生成する。結末が不明な項目はユーザーとの対話で深掘りし、日報へ反映する。

振り返り（KPT・学びの抽出・次アクションの言語化）は daily-review スキルの役割である。本スキルは「その日に何が起きたか」を時系列で可視化することに徹する。振り返り主体の指示を受けた場合は daily-review を使うこと。

## データソース

`scripts/collect.sh` が以下の3つを収集して stdout に出力する。

- **呟きログ**: `~/.guri3/ai/<yyyyMMdd>_nippo/tweets.jsonl`。UserPromptSubmit hook が `#t` プレフィックスの入力を蓄積する。1行の形式は `{"ts","session_id","cwd","text"}`
- **プロンプト履歴**: `~/.claude/history.jsonl`。全セッションのユーザープロンプトを時系列で保持する（AI の応答は含まない）
- **セッション**: `~/.claude/projects/<encodedPath>/<sessionId>.jsonl`。当日更新された主セッション（`agent-*` は除外）と、その `aiTitle`・`cwd`

## 手順

### 1. データ収集

`scripts/collect.sh` を実行して当日データを収集する。対象日の指定があれば引数として渡す。

```bash
scripts/collect.sh [対象日 YYYY-MM-DD]
```

- 引数を省略すると今日（ローカルタイム = JST）を対象にする
- 出力は `== TWEETS ==` / `== PROMPTS ==` / `== SESSIONS ==` の3セクション
- PROMPTS・TWEETS の時刻は JST。**SESSIONS の timestamp は UTC** である点に注意する（次のステップで JST に変換する）

### 2. タイムラインの構成

TWEETS / PROMPTS / SESSIONS を統合し、JST の時系列タイムラインを構成する。

- **セッションをレーンとして扱う**。各レーンは `aiTitle` と `cwd` 末尾のディレクトリ名で命名する（例: 「ログイン失敗時のアカウント凍結機能の調査（my-app）」）
- **SESSIONS の timestamp は UTC なので JST（+9 時間）に変換する**。PROMPTS・TWEETS は既に JST なのでそのまま使う
- **呟きは 💬 マーク付きのイベント**としてタイムライン上に配置する
- **PROMPTS はセッションの作業内容を推測する補助材料**として使う。history.jsonl には AI の応答が含まれないため、プロンプトの内容から「何をしていたか」を推測する
- SESSIONS の最初の timestamp が対象日より前になることがある（長時間・再開したセッションは mtime で当日に該当するため）。その場合はレーンの開始を対象日の稼働開始時刻に丸めるなど、当日分の可視化として自然になるよう扱う

### 3. 深掘り対話

タイムラインの中から**結末が不明な項目を2〜3個選び**、ユーザーに確認して回答を日報へ反映する。

深掘りの候補となる観点の例。

- 「ハマった」「詰まった」系の呟きが、その後解決したのか未解決なのか読み取れない
- 成果（コミット・PR・結論）が history から見えないセッション
- 呟きの内容とセッション履歴が食い違っている

AskUserQuestion または通常の対話で確認する。ユーザーが「深掘り不要」と言った場合はこのステップをスキップする。

### 4. 日報 md の生成

`~/.guri3/ai/<yyyyMMdd>_nippo/nippo.md` を生成する。

```bash
NIPPO_DIR="$HOME/.guri3/ai/$(date -j -f %Y-%m-%d "<対象日>" +%Y%m%d)_nippo"
mkdir -p "$NIPPO_DIR"
```

- **トリガー側から保存先が指定された場合はそれを優先する**（他スキルと同じ慣習）
- `.guri3` ディレクトリが存在しない場合は `ln -s $HOME/.guri3 .guri3` でシンボリックリンクを作成する

構成は以下とする。

```markdown
# 日報: YYYY-MM-DD (曜日)

## サマリ

（3行以内でその日の要点）

## タイムライン

（時刻・場所・内容の表。呟きは 💬 マークを付ける）

## 呟きログ

（その日の呟きを時系列で列挙）

## 深掘りメモ

（ステップ3で確認した内容と回答）

## 明日へ

（継続タスク・次にやること）
```

### 5. 可視化 HTML の生成

同ディレクトリに `timeline.html` を生成し、`open` で表示する。

- **自己完結型 HTML**（外部依存なし・インライン CSS・`file://` で直接開ける）にする
- **横軸 = 時刻**（当日の稼働時間帯）、**縦 = セッションレーン**、**呟きをマーカー**として配置するガントチャート風のタイムラインにする
- デザイントーンは `~/.claude/skills/morning-routine/assets/dashboard-example.html` を参照して合わせる（温かみのあるアイボリー基調・パネル・角丸・淡い影）
- 生成後に `open <timeline.html>` で表示する
- `mo` コマンドが存在すれば `mo --open <nippo.md>` も実行する。無ければスキップして保存先パスのみ報告する

### 6. 振り返りはやらない

KPT・学びの抽出・次アクションの言語化といった振り返りは本スキルの対象外である。それらは daily-review の役割なので、必要ならユーザーに daily-review を案内する。「明日へ」節は継続タスクの備忘であり、振り返りではない。

## 注意事項

- データが少ない日でも無理に内容を膨らませない。呟き・プロンプトから読み取れる事実に基づいて構成する
- ユーザーが知らない情報を推測で補わない。推測はあくまで「プロンプトからの作業内容の推測」に留め、断定しない
- collect.sh はエラーが出ても各セクションを可能な範囲で出力する。あるセクションが空でも他を使って日報を構成する
- コミット・PR は行わない
