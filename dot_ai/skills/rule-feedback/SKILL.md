# ルールフィードバック

セッション中のやり取りを振り返り、今後のセッションでも活かすべきルールや設定の変更を提案・適用する。

## 手順

### 1. セッションの振り返り

会話全体を振り返り、以下のパターンを探す:

- ユーザーが**繰り返し修正を求めた**こと（同じ種類の指摘が2回以上）
- ユーザーが**明確な好み**を示したこと（命名規則、コードスタイル、出力形式など）
- **暗黙の前提**として何度も適用されたルール（明文化されていないが一貫して従ったもの）
- 現在のルールと**矛盾する運用**が定着したこと

単発の指示や、そのセッション固有の文脈にしか当てはまらないものは除外する。「次のセッションでも同じことを言うか？」が判断基準。

### 2. 変更先ファイル別に提案を出力

以下のフォーマットで、変更先ファイルごとに提案をまとめる。該当がないファイルは省略してよい。

```
## 提案

### ~/.claude/CLAUDE.md（グローバルルール）
- [追加/変更/削除] 内容 → 理由: なぜルール化すべきか

### CLAUDE.md（プロジェクトルール）
- [追加/変更/削除] 内容 → 理由: なぜルール化すべきか

### CLAUDE.local.md（プロジェクト個人ルール）
- [追加/変更/削除] 内容 → 理由: なぜルール化すべきか

### settings.json / settings.local.json（Claude設定）
- [追加/変更/削除] 内容 → 理由: なぜルール化すべきか

### memory/（永続メモリ）
- [追加/変更/削除] 内容 → 理由: なぜルール化すべきか
```

### 3. ユーザーの承認を得る

提案を出力した後、ユーザーに確認を求める。ユーザーが承認した項目のみ適用する。部分的な承認（「1と3だけ」など）にも対応する。

### 4. 承認された変更を適用

承認を得たら、対象ファイルに変更を書き込む。

- CLAUDE.md / CLAUDE.local.md / ~/.claude/CLAUDE.md: 既存の構造に合わせてセクションを選び追記・編集する
- settings.json: 適切なキーに設定を追加・変更する
- memory/: メモリの書き込み手順（frontmatter付きmdファイル作成 → MEMORY.mdにインデックス追加）に従う

## 振り分けの判断基準

どのファイルに書くべきか迷ったときの指針:

| 条件 | 振り分け先 |
|------|-----------|
| どのプロジェクトでも当てはまる個人の好み | `~/.claude/CLAUDE.md` |
| このプロジェクト固有でチームに共有すべき | `CLAUDE.md` |
| このプロジェクト固有で個人的なもの | `CLAUDE.local.md` |
| 許可設定やフック、環境変数 | `settings.json` |
| ユーザーの役割・プロジェクト背景などのコンテキスト | `memory/` |
