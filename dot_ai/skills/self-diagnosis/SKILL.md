# Claude Code 自己診断

セッション横断の利用データを分析し、Claude Code の使い方そのものを改善する提案を出力する。

## 役割分担

| スキル | 対象範囲 | 入力 | 出力 |
|--------|---------|------|------|
| self-diagnosis（本スキル） | セッション横断（週次・月次） | 履歴・トランスクリプト・設定・スキル一覧 | 使い方の構造的な改善提案 |
| rule-feedback | 現在のセッション内 | 会話中の修正・好みのパターン | CLAUDE.md・settings・メモリへのルール追加 |
| daily-review | 当日1日 | GitHub・Slack などの活動履歴 | 仕事内容の振り返り（KPT） |

本スキルは「何を作ったか」ではなく「どう使っているか」を診断する。会話内容の詳細ではなく、利用パターンの統計と設定の整合性を見る。

## 手順

### 1. 分析期間の決定

デフォルトは直近30日。ユーザーが期間を指定した場合はそれに従う。

### 2. データ収集

集計スクリプトを実行する。

```bash
scripts/collect.sh <日数>
```

出力には日次アクティビティ・プロジェクト別プロンプト数・スラッシュコマンド手動起動回数・スキル起動回数・サブエージェント起動回数・モデル利用回数・ツール拒否回数・インストール済みスキル一覧が含まれる。

集計値だけで判断できない項目は追加調査する。

- 繰り返しの類似プロンプト: `~/.claude/history.jsonl` の `display` を期間で絞って眺め、同種の依頼が3回以上あればスキル化候補とする
- 拒否の内訳: トランスクリプトで拒否文言（`The user doesn't want to proceed with this tool use`）の前後を確認し、どのツール・コマンドが拒否されたかを特定する

### 3. 診断

以下の観点で分析する。データが薄い観点は「判断材料不足」と明記し、無理に提案をひねり出さない。

- **スキルの活用度**: インストール済みなのに期間内に一度も起動していないスキル（廃止・統合候補）。手動のスラッシュ起動ばかりで自動発火しないスキル（description の改善候補）。スキル化されていない繰り返しタスク（新規スキル候補）
- **権限設定**: 拒否が多いツール・コマンドのパターン。許可プロンプトが頻発しているなら fewer-permission-prompts の実行を提案する
- **ルールの整合性**: `~/.claude/CLAUDE.md` や各プロジェクトの CLAUDE.md のルールと実際の利用実態のずれ。セッション横断で繰り返されている修正指示
- **モデル委譲**: モデル利用の分布。上位モデルが単純作業に使われていないか、小タスクの委譲ルールが機能しているか
- **ワークフロー**: 特定プロジェクトへの偏り、サブエージェントの活用度、同種の調査を毎回ゼロからやり直していないか

### 4. レポート出力

診断結果を `.guri3/ai/<yyyyMMdd>_self_diagnosis/report.md` に保存し、ユーザーにレビューを依頼する。レポートの構成は次のとおり。

- 利用サマリー（期間・主要数値）
- 観点別の所見
- 改善提案（優先度順に、期待効果と適用先を添える）

トランスクリプトには業務内容が含まれるため、レポートは `.guri3` 以外に出力しない。引用する場合も固有名詞を含む部分は要約に留める。

### 5. 承認された提案の適用

ユーザーが承認した提案のみ適用する。適用は既存の仕組みに委譲する。

| 提案の種類 | 適用方法 |
|-----------|---------|
| 新規スキル作成 | write-a-skill スキル |
| スキルの description・内容改善 | dot_ai/skills/ 内を直接編集し、マージ後に apm-user-sync で反映 |
| ルール追加（CLAUDE.md・メモリ） | rule-feedback スキルの振り分け基準に従う |
| 権限の allowlist 追加 | fewer-permission-prompts または update-config スキル |
| フック・環境変数 | update-config スキル |

## 注意事項

- `stats-cache.json` の `lastComputedDate` が古い場合、日次アクティビティは実態より少なく出る。他の集計値を主とする
- モデル利用回数は期間内に更新されたトランスクリプト全体を数えるため、期間開始前から続くセッションの分を含む概算値である
