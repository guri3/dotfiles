# 問題解決ワークフロー（solve）

## 概要

Issueやユーザーの要望を起点に、事前調査→計画→設計→実装→レビューの全フェーズを一貫して進める。計画フェーズと設計フェーズのそれぞれでユーザーの承認を得てから次に進むことで、手戻りを防ぐ。

## 入力

以下のいずれか:

- GitHub Issue URL（例: `https://github.com/owner/repo/issues/123`）
- ユーザーの要望・問題説明（自然言語）

## 出力先ディレクトリ

すべての成果物は `.guri3/ai/<作業名>/` に保存する。

- Issue起点の場合: `.guri3/ai/<yyyyMMdd>_<issue_number>_<issue_slug>/`
- 要望起点の場合: `.guri3/ai/<yyyyMMdd>_<作業スラッグ>/`
- `<yyyyMMdd>` は作業開始日の日付（例: `20260305`）
- ディレクトリが存在しない場合は自動作成する
- `.guri3` ディレクトリが存在しない場合はシンボリックリンクを作成する: `ln -s $HOME/.guri3 .guri3`

## ワークフロー

### フェーズ1: 問題の把握と事前調査 → plan.md作成

解決すべき問題を理解し、大まかな方針を固める。

**Issue起点の場合**:

```bash
gh issue view <issue_number> --repo <owner/repo> \
  --json number,title,body,labels,assignees,milestone,comments
```

**調査内容**:

- 問題・要望の背景と目的を理解する
- 関連するコードやファイルを調査する
- 技術的な制約・前提条件を洗い出す
- 解決アプローチの候補を複数検討する（自分で選ばず、複数の選択肢をユーザーに提示する）
- タスクの規模を見極め、Sub Issuesへの分割が適切かどうか判断する
- リスクや注意点を整理する

**plan.mdの構成**:

```markdown
# 計画

## 問題の概要

<何を解決するか>

## 調査結果

<調査で把握したこと・関連コード・技術的制約>

## 解決方針の選択肢

### 案A: <アプローチ名>

<概要・メリット・デメリット>

### 案B: <アプローチ名>

<概要・メリット・デメリット>

（案が1つしかない場合はその旨を明記する）

## Sub Issue提案

（以下の基準に該当する場合のみ記載。該当しない場合はこのセクションを省略する）

以下の観点でタスクを評価し、分割が望ましい場合は提案する。

- 独立してリリース・検証できる機能単位が複数ある
- 担当者や優先度が異なる作業が混在している
- 1つのIssueとして管理するには広すぎて追跡が困難

**提案するSub Issue**:

- [ ] <Sub Issue 1のタイトルと概要>
- [ ] <Sub Issue 2のタイトルと概要>

このsolveの対象をどのスコープにするかユーザーに確認する。

## 実装スコープ

<今回対応する範囲と対応しない範囲>

## リスク・注意点

<実装時に注意すべき点>
```

出力先: `.guri3/ai/<作業名>/plan.md`

### フェーズ2: plan.mdのレビュー依頼

plan.mdが生成されたらユーザーに確認を求める。

```
解決計画（plan.md）を作成しました。

ファイル: .guri3/ai/<作業名>/plan.md

解決方針の選択肢を複数提示しています。どの案で進めるかご選択ください。
修正や追加の案が必要な場合はその旨をお伝えください。
```

**ユーザーの承認なしに次フェーズへ進まないこと。**

ユーザーの承認が得られたら、Issue起点の場合はplan.mdの内容をIssueにコメントとして投稿する:

```bash
plan_comment_url=$(gh api repos/<owner>/<repo>/issues/<issue_number>/comments \
  --method POST \
  --field body="$(cat .guri3/ai/<dir>/plan.md)" | jq -r '.html_url')
echo "計画コメントURL: $plan_comment_url"
```

### フェーズ3: 詳細設計 → design.md作成

plan.mdを踏まえ、実装の詳細を設計する。

**design.mdの構成**:

```markdown
# 詳細設計

## 実装方針

<具体的な実装のアプローチ>

## 変更対象ファイル

<変更・作成するファイルとその役割>

## 実装詳細

<各ファイルへの変更内容・インターフェース設計・データ構造など>

## テスト方針

<何をどのようにテストするか・テストケースの概要>

## PR分割方針

実装をどの粒度でPRに分割するか整理する。PRは独立してレビュー・マージできる単位を基本とする。

| PR | 内容 | ベースブランチ |
|----|------|--------------|
| PR-1 | <内容> | main |
| PR-2 | <内容> | main or PR-1ブランチ |

## PR依存関係

PRの依存関係を明示する。並行実装できるものとシーケンシャルに進める必要があるものを区別する。

- **並行実装可能**: <PR番号のリスト>（互いに依存しないため同時に進められる）
- **直列実装が必要**: <PR番号と順序>（例: PR-1完了後にPR-2を開始）
```

出力先: `.guri3/ai/<作業名>/design.md`

### フェーズ4: design.mdのレビュー依頼

design.mdが生成されたらユーザーに確認を求める。

```
詳細設計（design.md）を作成しました。

ファイル: .guri3/ai/<作業名>/design.md

内容をご確認ください。問題なければ実装を開始します。
修正が必要な場合はその旨をお伝えください。
```

**ユーザーの承認なしに次フェーズへ進まないこと。**

ユーザーの承認が得られたら、Issue起点の場合はdesign.mdの内容をIssueにコメントとして投稿する:

```bash
design_comment_url=$(gh api repos/<owner>/<repo>/issues/<issue_number>/comments \
  --method POST \
  --field body="$(cat .guri3/ai/<dir>/design.md)" | jq -r '.html_url')
echo "設計コメントURL: $design_comment_url"
```

### フェーズ5: 実装開始（ブランチ・空コミット・Draft PR）

design.mdの承認後、PR分割方針に従ってブランチを作成し、空コミットとDraft PRで実装の意図を宣言する。

**PRが1つの場合**:

```bash
# ブランチ作成
git checkout -b <ブランチ名>

# 空コミットとPush
git commit --allow-empty -m "WIP: <実装内容の簡潔な説明>"
git push -u origin <ブランチ名>

# Draft PR作成
gh pr create --draft \
  --title "<PRタイトル>" \
  --body "..." \
  --base <default_branch> \
  --assignee @me
```

**PRが複数の場合（並行実装あり）**:

各PRに対してworktreeを作成し、独立した作業ディレクトリで並行実装できるようにする。

```bash
# 並行実装するPRごとにworktreeを作成
git worktree add ../<リポジトリ名>-<ブランチ名> -b <ブランチ名>

# 各worktreeで空コミットとPush
cd ../<リポジトリ名>-<ブランチ名>
git commit --allow-empty -m "WIP: <PR内容の説明>"
git push -u origin <ブランチ名>

# Draft PR作成（各ブランチ）
gh pr create --draft \
  --title "<PRタイトル>" \
  --body "..." \
  --base <依存するブランチ or default_branch> \
  --assignee @me
```

依存関係のあるPRは依存先のブランチをベースブランチとして設定する。

- `.github/PULL_REQUEST_TEMPLATE.md` が存在する場合は必ずそのテンプレートに準拠する
- ベースブランチは `gh repo view --json defaultBranchRef` で確認する
- Issue起点で最終的にIssueをクローズするPRには `Closes #<issue_number>` を含める

### フェーズ6: TDD実装

design.mdのPR依存関係に従い、並行実装できるものはサブエージェントを同時に起動して実装する。

**並行実装できるPRがある場合**:

- 各worktreeに対して独立した `tdd-implementer` サブエージェントを同時に起動する
- 各サブエージェントには担当PRの実装詳細・worktreeのパス・ブランチ名を伝える

**直列実装が必要なPRがある場合**:

- 依存先のPRの実装が完了してから次のPRのサブエージェントを起動する

**各サブエージェントへの共通指示**:

- design.mdの実装詳細と担当範囲を伝える
- ローカルでテストが実行できる場合はテスト結果を元に改善を繰り返す
- CIが整っている場合はCIの結果も確認する

**TDDサイクル**:

1. テストを先に書く（Red）
2. テストを通す最小限の実装をする（Green）
3. コードを整理する（Refactor）

### フェーズ7: セルフレビュー

実装が完了したら、コードを自身でレビューする。

**チェック内容**:

- design.mdの要件をすべて満たしているか
- テストが適切に書かれているか（漏れ・過不足がないか）
- セキュリティ上の問題がないか（XSS・SQLインジェクション・OWASP Top 10）
- 不要なコードや過剰な実装がないか
- ファイル末尾の改行など規約に沿っているか

修正が可能なものは自身で修正する。設計の根本的な見直しが必要な場合はユーザーに相談する。

### フェーズ8: ユーザーへのレビュー依頼

一通りの実装とセルフレビューが完了したら、ユーザーにレビューを依頼する。

```
実装が完了しました。レビューをお願いします。

## 実装内容
- <変更点1>
- <変更点2>

## テスト状況
- <テストの実行結果>

## PR
<PR URL>

## 確認してほしい点
- <特に確認してほしいポイント>
```

## 重要な原則

### ユーザーとの協調

- plan.mdとdesign.mdはそれぞれユーザーの承認を得てから次フェーズへ進む
- 修正依頼があった場合は修正後に再度確認を求める
- 設計の根本的な問題に気づいた場合は実装を止めてユーザーに相談する

### Issue起点の場合のGitHub連携

- plan.md・design.mdはユーザーの承認後にIssueコメントとして投稿する
- 承認前に投稿しない

### 実装の品質

- 過剰な実装をしない。要求されていない機能は追加しない
- セルフレビューは形式的に行うのではなく、実際に問題を見つける意識で行う
- テストはTDDで先に書くことを守る

### エラーハンドリング

- `gh` コマンドが失敗した場合はエラー内容をユーザーに伝えて対処を求める
- ブランチが既に存在する場合はユーザーに確認してから処理を継続する
- テストやCIが失敗した場合は原因を調査して修正する。同じ失敗を繰り返さない

### ドキュメントの品質

- すべてのドキュメントはAGENTS.mdのドキュメント哲学に従う（「だ。」「である。」調）
- ファイル末尾に改行を含める
