# PR分割計画スキル（split-pr）

## なぜこのスキルがあるのか

大きな機能を実装するとき、PRを小さく分けると以下のメリットがある。

- レビュアーの負荷が下がる
- mergeまでのリードタイムが短くなる
- 変更の意図が明確になる

一方、分割数が増えると以下の課題が出る。

- ブランチ切り替え時のtypo（`adjust-feat-desing-1` のような）
- 「この変更はどのPRで受けるか」を実装中に悩んで再整理が発生
- 派生元ブランチの取り違え、未pushでの切り替え
- 命名（概念名）が実装途中で揺れる

これらの課題は**着手前に「誰が・何を・どのブランチで受けるか」を紙に書き出しておく**ことで大半が防げる。本スキルはその計画表を作る。

## いつ使うか

- 大きな機能・リファクタリングに取り組む前
- 既に書き始めた変更を複数PRに分割し直したいとき
- 派生ブランチ（stacked PR）の関係が複雑になってきたとき
- レビュー依頼中PRが増えてきて、次の依存PRをどこから切るか迷ったとき

## 入力

以下のいずれか。

- 実装したい機能・タスクの説明（自然言語）
- 現在の作業ブランチで蓄積している変更（`git status` / `git diff` で読み取る）
- GitHub Issue URL

## 出力先

計画は `.guri3/ai/<yyyyMMdd>_<作業スラッグ>/pr_split_plan.md` に保存する。

- `<yyyyMMdd>` は作業開始日
- 既に `solve` スキルなどで作業ディレクトリが作られている場合はそこに `pr_split_plan.md` を追加する
- `.guri3` ディレクトリが存在しない場合はシンボリックリンクを作成: `ln -s $HOME/.guri3 .guri3`

## 手順

### 1. コンテキストを掴む

- 現在のブランチとベースブランチを確認: `git status`, `git log --oneline <base>..HEAD`
- 未コミット・未pushの変更を把握: `git status`, `git diff`, `git log @{upstream}..HEAD`
- リポジトリのデフォルトブランチを把握: `gh repo view --json defaultBranchRef`
- 既に作ったPRがあればそれもリスト: `gh pr list --author @me --state all --limit 10`

これらを省略すると「今どこから切れば良いか」の判断を誤る。必ず実施すること。

### 2. 変更を意味単位に分解する

ユーザーの説明と `git diff` から、変更を以下の観点で分解する。

- **独立性**: この変更単体でmergeしてもシステムが壊れないか
- **レビュー容易性**: 1人のレビュアーが30分以内でレビューできる粒度か
- **依存関係**: 他の変更に依存するか、他から依存されるか
- **リスク**: 影響範囲が大きい変更（命名変更、API変更、共通コンポーネント変更）は単独PRにする

よくある分解軸の例:

- 基盤系（CI・ビルド設定・lint）を先に切り出す
- デザイントークン・定数追加などの「足場」を先に出す
- コンポーネント/View単位で分ける（画面Aと画面Bを別PRに）
- 命名変更・リネームは単独PR（巻き込むとレビューが困難になる）
- 文言更新・ライセンス追記など軽微なものも単独PR

### 3. 分割計画表を作る

以下のテンプレートを `pr_split_plan.md` に書き込む。

```markdown
# PR分割計画: <タスク名>

- 作業開始日: <yyyy-MM-dd>
- ベースブランチ: <main / master>
- 関連Issue/親PR: <URL>

## 計画表

| # | 目的 | ブランチ名 | 派生元 | 含める変更 | 状態 |
|---|------|------------|--------|------------|------|
| 1 | CIでのビルド検証を追加 | `ci-add-build-check` | `main` | ワークフロー・スキーム定義の追加 | open / merged |
| 2 | デザイントークン追加 | `add-design-tokens` | `main` | 共通色/余白定数の追加 | open |
| 3 | 画面Aのデザイン調整 | `adjust-view-a` | 2 | 画面A関連ファイルのみ | open |
| ... | ... | ... | ... | ... | ... |

## 依存関係

```
main
 ├── 1 (ci-add-build-check)
 └── 2 (add-design-tokens)
     ├── 3 (adjust-view-a)
     └── 4 (adjust-view-b)
         └── 5 (adjust-view-c)
```

## 命名チェックリスト

- [ ] ブランチ名に `/` を使っていない（区切りは `-`）
- [ ] ブランチ名にtypoがない（実装前に声に出して読む）
- [ ] 変更の中心概念の名前が仕様と一致している（例: `Period` か `EndDate` か）

## メモ

- リリースPR: <URL>
- 作業中に気づいたことはここに追記
```

### 4. 計画のレビューを依頼する

計画表を作ったらユーザーに確認を求める。

```
PR分割計画を作成しました。

ファイル: .guri3/ai/<作業名>/pr_split_plan.md

計画表と依存関係を提示しています。以下の観点で確認してください:

- 粒度は適切か（小さすぎる / 大きすぎる ものはないか）
- 依存関係は正しいか
- 命名は仕様と揃っているか
- 派生元ブランチの指定は妥当か

修正が必要な場合はお伝えください。このまま進めて良ければ、最初のPRから着手します。
```

**ユーザー承認なしに実装へ進まないこと**。ここで粒度を議論するのが本スキルの価値。

### 5. 進行中の更新

実装中、状況が変わったら計画表を更新する（mergedに変わった、新しいPRを追加する、命名が変わった、など）。

- 新しいブランチを切る前に必ず計画表を確認し、派生元が正しいか検証する
- ブランチ切り替え前に `git status` で未push/未コミットがないかチェックする
- 実装中に中心概念の命名変更が発生したら、該当PRの「含める変更」欄と「メモ」を更新する

## 既存スキルとの関係

- **solve**: `solve` のフェーズ3（design.md作成）でPR分割方針が決まる場合がある。本スキルはそれを「作業用の生きた計画表」として拡張・更新する役割
- **create-pr**: 各PR作成時に呼び出す。計画表の1行が1つのPRに対応する
- **git-commit**: 各ブランチでのコミット作成に使用

## ブランチ命名のルール（AGENTS.md準拠）

- 区切り文字は `-`（`/`は使わない）
  - 例: `adjust-view-a`（良い）
  - 例: `feat/adjust-view` ❌ / `adjust/view` ❌
- typo防止のため、計画表に書いた名前をコピペして使う
- 連番は `-1`, `-2` のように末尾に付ける（派生順が明確になる）

## 失敗パターンと回避策

- **「計画を作ったのに参照しない」**: 計画表は実装中も開いたままにする。ブランチを切るたびに計画表の「ブランチ名」列をコピペする
- **「計画が古くなる」**: PRをopen/mergedに変えるたびに状態列を更新する。更新しないと次の派生元判断を誤る
- **「粒度を細かくしすぎる」**: 3行変えるだけのPRが3連続で並んでいるなら、それは計画の失敗。意味単位で括り直す
- **「依存関係を深くしすぎる」**: stackが4段以上になったら、どこかで並列化できないか見直す。深いstackはrebase地獄になる
